Assembly Language Game Code Explanation

Game: Kaun Banay Ga Crore Patti (Who Wants to Be a Millionaire)
Language: x86 Assembly (MASM with Irvine32 library)
Purpose: This document explains each section of the assembly code in detail.

Section 1: Header and Library Includes

INCLUDE Irvine32.inc

Line 1: INCLUDE Irvine32.inc

What it does: This directive includes the Irvine32 library, which provides useful functions for input/output operations in assembly language.

Why it's used: The Irvine32 library contains pre-written procedures like:

  • WriteString - to display text
  • ReadString - to read user input
  • ClrScr - to clear the screen
  • Crlf - to print a new line
  • exit - to exit the program

How it works: The assembler copies all the code from Irvine32.inc into this file during compilation, making all those functions available to use.

Section 2: Data Section - Title and ASCII Art

.data title1 BYTE " *** WELCOME TO KAUN BANAY GA CRORE PATTI *** ",0

Line 4: .data

What it does: Marks the beginning of the data section where all variables and constants are declared.

Why it's used: In assembly, you need to separate data (variables) from code (instructions). The .data section is where you store all your strings, numbers, and arrays.

Line 6: title1 BYTE " *** WELCOME TO KAUN BANAY GA CRORE PATTI *** ",0

What it does: Creates a string variable called title1 that contains the game's welcome message with decorative asterisks and centered formatting.

Why it's used: This is the title that will be displayed at the start of the game. The asterisks and spacing create a visually appealing header.

How it works:

  • title1 - the name of the variable
  • BYTE - declares it as a byte array (each character is 1 byte)
  • The text in quotes - the actual string content with formatting spaces and asterisks
  • ,0 - null terminator (marks the end of the string, required for string functions)

;Drawing of amitab bachan L1 BYTE " ",0 L2 BYTE " ++==+++=**+++* ",0 ... (lines 8-47 contain ASCII art lines L1 through L39)

Lines 8-47: ASCII Art Drawing

What it does: Creates 39 separate string variables (L1 through L39), each containing one line of ASCII art that forms a picture of Amitabh Bachchan.

Why it's used: This creates a visual element for the game, making it more engaging. Each line (L1, L2, L3, etc.) represents one row of the ASCII art.

How it works:

  • Each L# BYTE "..." creates a string variable
  • The characters (+, *, #, @, etc.) form the picture when printed line by line
  • All lines end with ,0 (null terminator)

drawing DWORD OFFSET L1, OFFSET L2, OFFSET L3, ..., OFFSET L39

Line 49: drawing Array

What it does: Creates an array called drawing that stores the memory addresses (offsets) of all 39 ASCII art lines.

Why it's used: Instead of manually printing each line, we can loop through this array to print all lines automatically.

How it works:

  • DWORD - each element is 4 bytes (32-bit address)
  • OFFSET L1 - gets the memory address where L1 is stored
  • The array contains 39 addresses, one for each line
  • We can access drawing[0] to get L1's address, drawing[1] for L2, etc.

Lines 51-59: Question 1 Structure (Box Format)

What it does: Defines the first question in a formatted box display with borders made of underscores and pipes.

Why it's used: The box format makes the questions visually appealing and easier to read, similar to a TV game show display.

How it works:

  • Q1 - top border line (underscores)
  • A1 - question header line ("QUESTION 1")
  • B1 - the actual question text
  • C1 - empty/spacing line
  • D1 - first row of options (A and C side by side)
  • E1 - second row of options (B and D side by side)
  • G1 - lifeline option line
  • F1 - bottom border line (underscores)
  • All are BYTE arrays (strings) ending with ,0

Note: Questions 2-5 (lines 61-83) reuse Q1, F1, C1, and G1 for borders/formatting, and only define their unique content (A2-A5 for headers, B2-B5 for questions, D2-D5 and E2-E5 for options).

Lines 85-90: Question Arrays

What it does: Creates arrays for each question that store the addresses of all 8 display lines needed to render the question box.

Why it's used: This allows the program to loop through and display all 8 lines of the question box format automatically.

How it works:

  • Question1[0] = address of Q1 (top border)
  • Question1[1] = address of A1 (question header)
  • Question1[2] = address of B1 (question text)
  • Question1[3] = address of C1 (empty line)
  • Question1[4] = address of D1 (first option row: A and C)
  • Question1[5] = address of E1 (second option row: B and D)
  • Question1[6] = address of G1 (lifeline option)
  • Question1[7] = address of F1 (bottom border)

Note that Questions 2-5 share Q1 (top border), C1 (empty line), G1 (lifeline option), and F1 (bottom border) for consistency.

Lines 92-97: 50:50 Lifeline Question Arrays

What it does: Creates special versions of each question array for the 50:50 lifeline, which removes two wrong answers.

Why it's used: When the 50:50 lifeline is used, the question needs to be displayed with only 2 options (one correct, one wrong) instead of 4.

How it works:

  • These arrays have the same structure as regular Question arrays (8 elements)
  • They use OFFSET C1 (empty line) to replace the options that should be hidden
  • For example, Question1not hides option A by replacing D1 with C1 (empty), showing only B and D options
  • The correct answer option (E1 in Question1not) is always kept visible
  • One wrong option is also kept visible to maintain the challenge

What it does: Creates two variables (one for each lifeline) that track whether each lifeline has been used.

Why it's used: Each lifeline can only be used once. These variables store 0 (not used) or 1 (used) to prevent reusing lifelines.
